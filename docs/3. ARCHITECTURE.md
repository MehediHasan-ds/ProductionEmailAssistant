# Architecture. Stages 8 and 9 of the ASDL

This file is the engineering shape of the Email Generation Assistant. It sits directly on top of the analysis I wrote in the ASDL document, so nothing here is invented from preference. Every component exists because an earlier stage of the lifecycle forced it.

## How I arrived at this architecture

Before describing what I am building, I want to be explicit about why, because the lifecycle is the whole point.

- The **root cause analysis** showed that the real problem is simultaneous constraint satisfaction without an enforced review pass. That single conclusion **forces** a separate evaluation step and a refinement step. The agent exists to review, not merely to write.
- The **feasibility assessment** said rules are insufficient but a swarm of agents is unjustified. That **forces** a deliberately minimal design, one agent with a self refinement loop.
- The **requirements** set hard targets: under twenty seconds including retries, under one cent per email, and reproducible evaluation at temperature zero. Those **force** the choices around cost, latency, and determinism.
- The **systems analysis** said every dependency can fail. That **forces** provider fallback and graceful degradation of the semantic metric.

So this architecture is the direct, traceable consequence of the analysis, not a set of fashionable choices.

## The components I am building

I am organizing the system into a small number of components, each with one narrow responsibility.

**Presentation layer.** A single page web interface where a user enters the intent, the facts, and the tone, picks a provider, and hits generate. It shows the produced email, the score breakdown, and the trace of refinement attempts. It is plain HTML and JavaScript with no build step, because the value is in the agent, not the shell around it.

**API layer.** A FastAPI service exposing a handful of endpoints. Generate runs the full agent loop. Evaluate scores one email without retry. Run Evals executes the whole evaluation suite and produces a report. Health and Providers support basic operations. This layer only validates input, calls the agent core, and shapes responses. It holds no business logic.

**Agent core.** The heart of the system, the generate, evaluate, refine loop. It coordinates three subcomponents.

**The generator.** Produces a draft email from the three inputs using advanced prompting. I am combining four techniques and documenting them. **Role playing** gives the model a senior business communications writer persona. **Few shot** injects two curated examples of intent, facts, and tone mapped to ideal emails. **Chain of thought** forces a structured JSON response containing a reasoning field, a subject, and a body, so the model plans structure and tone markers and weaves each fact deliberately before writing. **Self refinement** appends the evaluator's critique into the prompt on retry. The reasoning field is also inspectable and gradeable, which is a bonus for evaluation.

**The evaluator.** Scores a draft using a hybrid approach, because no single measure captures email quality. It runs three groups of metrics. **Rule based, deterministic, no extra LLM cost**: fact coverage, tone match, structure compliance, length appropriateness, readability, placeholder leak, hallucination flag, and redundancy. **An LLM as judge, rubric scored, blind to the reference**: tone fidelity, fact integration, professionalism, clarity and coherence, intent alignment, and overall send readiness. **Reference comparison against the human reference email**: ROUGE L, BLEU, and Jina cosine similarity. All of this aggregates into a weighted score out of one hundred with a configurable pass threshold.

**The refinement controller.** Looks at the score. If the draft meets the threshold, it stops. If not, it builds a textual critique from the low scoring dimensions and asks the generator to revise, up to a capped number of attempts, keeping the best draft by score. This is the enforced review pass the root cause demanded.

**The provider agnostic LLM client.** A single client that speaks to OpenAI, DeepSeek, and Gemini, all of which expose OpenAI compatible APIs. Both the generator and the judge use this one client. The provider is selectable per request, with a sensible default from configuration. This is what lets the evaluation suite compare all three providers on identical ground.

**The local embedding engine.** The semantic similarity metric uses my local quantized Jina embeddings version five model, run through the official optimum plus torch path on CPU only. It loads the quantized ONNX file once as a singleton at startup. It prefixes both the candidate and the reference email with the document marker, since I am comparing two emails symmetrically and this retrieval tuned model expects that. It does last token pooling and L2 normalization, the two steps the official snippet omits but the model pipeline requires. This keeps the semantic metric provider agnostic, offline, and free of per call cost, which matters because one of the providers has no embeddings endpoint at all.

## How a request moves through the system

The lifecycle of a single generation request is as follows.

1. The user submits intent, facts, tone, and an optional provider to the generate endpoint.
2. The API layer validates the input and hands it to the agent core.
3. The generator builds the prompt with role playing, few shot examples, and a chain of thought structure, and calls the provider client.
4. The generator returns a structured JSON with reasoning, subject, and body.
5. The evaluator scores the draft across all three metric groups and produces the weighted total.
6. If the total is below the threshold and attempts remain, the refinement controller builds a critique, the generator revises, and the loop repeats, capped at three attempts.
7. The agent core returns the best email, the per dimension scores, the number of attempts, and the full critique trace.
8. The API layer returns this to the interface, which renders the email, the score breakdown, and the trace.

## Alternative paths I considered and rejected

I did not land on this design by default. I weighed other technically valid approaches and turned them down for specific reasons.

**Alternative one. Single shot generation with no refinement loop.** Call the model once with a strong prompt and return whatever it produces. This is the simplest possible design and the fastest to build. I rejected it because it removes the enforced review pass, which is the entire reason the agent exists according to my root cause analysis. It also fails the stated requirement for retry with dynamic evaluation, and one shot outputs are exactly the error prone drafts a human already produces unaided. The refinement loop is the source of measurable quality lift, and I will prove that lift by comparing a no retry baseline against the full loop in the evaluation report.

**Alternative two. A multi agent framework with a planner, a drafter, and a critic.** Use an orchestration framework to spin up several cooperating agents. This sounds more powerful and is a common reflex. I rejected it because the feasibility stage said to introduce the smallest amount of agent the problem requires, and this problem needs reasoning plus review, not a team. A swarm multiplies latency and cost, introduces handoff and infinite loop failure modes that the systems analysis warned about, and adds heavy dependencies and non determinism that fight the reproducibility requirement. A custom loop is more controllable, cheaper, easier to debug, and easier to keep deterministic at temperature zero.

**Alternative three. Fine tuning a small model on email data.** Train or fine tune a dedicated model to write professional emails. This would feel like a more permanent solution. I rejected it because I do not have the volume of labeled, high quality email data that fine tuning demands, because a fine tuned model locks me to one provider and fights the switchability requirement, and because tone and intent variety across the scenarios would require constant retraining. Prompt engineering with a refinement loop achieves the quality gains far faster, with no data debt, and stays provider agnostic. Fine tuning is the right tool only if prompting plateaus, which the evaluation will tell me.

**Two shorter mentions.** I also rejected a pure template plus slot fill design, because rules and templates cannot handle tone nuance or natural fact weaving, which the feasibility stage already settled. And I rejected using a hosted embeddings API for the semantic metric, because one of my three providers offers no embeddings endpoint, so a hosted metric would not be provider agnostic, would add per call cost and a network dependency, and would duplicate a capability I already have locally for free.

## How I engineered to the non functional requirements

- **Latency.** The loop is capped at three attempts, the embedding model loads once at startup, and short emails use a short max length so CPU inference stays fast. Generation including retries targets under twenty seconds.
- **Cost.** Most metric work is rule based and free. The LLM judge and generator are the only paid calls, capped per request, targeting under one cent per email.
- **Reproducibility.** The generator and the judge run at temperature zero during evaluation so results are comparable across runs and across providers.
- **Resilience.** The provider client retries then falls back to another provider on timeout, rate limiting, or quota errors. If the embedding engine fails at runtime, the evaluator degrades the semantic metric to lexical only rather than failing the whole request.
- **Safety boundary.** The agent drafts and scores and refines. It never sends. It never calls external systems. Low confidence or contradictory input returns the raw draft with a flag rather than inventing facts.

## Build time safety checks I will run first

Before this architecture is wired together, I will confirm two things about the embedding engine, because they are the easiest to get silently wrong. I will verify the BOS token is present at the start of the tokenized output, since the EuroBERT model expects it. And I will run a cosine sanity check that related phrases score high and unrelated phrases score low, because if everything scores near one it means the pooling is wrong, which is the failure mode that would quietly corrupt the semantic metric.
